# User Story - US_089

## Story ID

- ID: US_089

## Story Title

- Tiered Retention and Backup Encryption

## Description

- As a security officer, I want all backups encrypted with AES-256, stored in geographically separate locations, and subject to quarterly restoration testing, so that backup data is secure and verifiably recoverable.

## Acceptance Criteria

1. **Given** a backup is created, **When** it is stored, **Then** the backup file is encrypted with AES-256 before being written to the storage location.
2. **Given** encrypted backups exist, **When** they are stored, **Then** a copy is maintained in a geographically separate storage location from the primary database.
3. **Given** the quarterly testing schedule, **When** a restoration test is due, **Then** the system provides documented procedures and the admin can restore the latest backup to a test environment and verify data integrity.
4. **Given** a restoration test completes, **When** the results are verified, **Then** the system validates row counts, checksums, and referential integrity against the source backup.

## Edge Cases

- What happens when the geographic backup transfer fails? System retries 3 times with exponential backoff; if all fail, the primary backup is preserved and an alert is sent.
- How does the system handle encryption key management for backups? Encryption keys are stored separately from backups using the system's key management configuration; key loss renders backups unrecoverable.
