# User Story - US_037

## Story ID

- ID: US_037

## Story Title

- Notification Delivery Logging and Retry

## Description

- As a system administrator, I want all notification delivery attempts logged with status tracking and failed deliveries retried with exponential backoff, so that I can monitor notification reliability and ensure patients receive communications.

## Acceptance Criteria

1. **Given** any notification is sent (email or SMS), **When** the delivery completes, **Then** the system logs the attempt in NotificationLog with status (sent, failed, bounced), timestamp, channel, and recipient.
2. **Given** a notification delivery fails, **When** the failure is detected, **Then** the system retries up to 3 times with exponential backoff (1 min, 5 min, 15 min).
3. **Given** all 3 retry attempts fail, **When** the final retry fails, **Then** the system marks the notification as "permanently_failed" and flags it for staff review.
4. **Given** an admin views the notification log, **When** they filter by status, **Then** the log displays delivery statistics including success rate, failure rate, and average delivery time.

## Edge Cases

- What happens when the logging service itself is unavailable? System queues log entries in memory (max 1000) and flushes when logging recovers.
- How are bounced emails handled differently from failed emails? Bounced emails trigger contact validation flag on the patient record; failed emails only trigger retries.
