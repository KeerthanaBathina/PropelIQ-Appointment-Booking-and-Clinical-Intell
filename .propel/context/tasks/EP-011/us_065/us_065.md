# User Story - US_065

## Story ID

- ID: US_065

## Story Title

- Session Security Hardening

## Description

- As a security engineer, I want automatic 15-minute session timeout, concurrent session prevention, and account lockout enforcement, so that unauthorized access through idle sessions, session hijacking, or brute-force attacks is prevented.

## Acceptance Criteria

1. **Given** a user is authenticated, **When** 15 minutes of inactivity pass, **Then** the session is automatically invalidated and the user is redirected to the login page.
2. **Given** a user is logged in on Device A, **When** they attempt to log in on Device B, **Then** the session on Device A is terminated and the user is notified of concurrent session prevention.
3. **Given** a user account exists, **When** 5 consecutive failed login attempts occur, **Then** the account is locked for 30 minutes and an audit log entry records the lockout event.
4. **Given** a session timeout is approaching, **When** 2 minutes remain, **Then** a modal countdown with "Extend Session" button is displayed to the user.

## Edge Cases

- What happens when the session extension request fails due to network issues? System invalidates the session on timeout; next page load redirects to login with "Session expired" message.
- How does the system handle lockout for admin accounts? Same 5-attempt lockout applies; a separate admin recovery process (database-level unlock) is documented for emergencies.
