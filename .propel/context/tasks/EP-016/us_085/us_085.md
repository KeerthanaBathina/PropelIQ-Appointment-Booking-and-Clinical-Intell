# User Story - US_085

## Story ID

- ID: US_085

## Story Title

- Data Validation Rules

## Description

- As a developer, I want application-level data validation enforcing date ranges, email format, duplicate prevention, and code library validation, so that invalid data is rejected at the system boundary before reaching the database.

## Acceptance Criteria

1. **Given** an appointment booking is submitted, **When** the date is validated, **Then** the system rejects dates beyond 90 days from today with a descriptive error message.
2. **Given** a patient email is submitted, **When** format validation runs, **Then** the system validates against a standard email regex pattern and rejects invalid formats.
3. **Given** a booking request is submitted, **When** duplicate checking runs, **Then** the system enforces the patient_id + appointment_time unique constraint and returns a clear error for duplicates.
4. **Given** ICD-10/CPT codes are entered, **When** code validation runs, **Then** the system validates codes against the current code library and rejects invalid codes with suggested alternatives.

## Edge Cases

- What happens when a validation rule is updated while requests are in-flight? Validation rules are loaded at application startup and refreshed on configuration change; in-flight requests use the rules at the time of processing.
- How does the system handle timezone edge cases for date validation (booking at 11:59 PM)? All dates are normalized to the clinic's local timezone before validation.
