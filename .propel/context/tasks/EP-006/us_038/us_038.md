# User Story - US_038

## Story ID

- ID: US_038

## Story Title

- Clinical Document Upload and Validation

## Description

- As a staff member, I want to upload clinical documents (PDF, DOCX, TXT, PNG, JPG) with format and size validation and AES-256 encrypted storage, so that patient records are securely stored and ready for AI parsing.

## Acceptance Criteria

1. **Given** a staff member is on the Document Upload screen, **When** they select or drag-and-drop a file, **Then** the system validates the format (PDF, DOCX, TXT, PNG, JPG) and size (max 10MB) before upload begins.
2. **Given** the document passes validation, **When** the upload completes, **Then** the file is stored with AES-256 encryption at rest and a ClinicalDocument record is created with status "uploaded."
3. **Given** the staff member uploads a document, **When** they assign a category (lab result, prescription, clinical note, imaging report), **Then** the category is saved to the ClinicalDocument record.
4. **Given** the upload succeeds, **When** the operation completes, **Then** the source document attribution (filename, upload timestamp, uploading user) is recorded.
5. **Given** a file exceeds 10MB or has an unsupported format, **When** the upload is attempted, **Then** the system rejects it with an error listing supported formats and size limits.

## Edge Cases

- What happens when the upload is interrupted mid-transfer? System discards partial upload, displays retry option, and does not create a ClinicalDocument record.
- How does the system handle a file with a valid extension but corrupt content? System accepts the upload but flags parsing errors during the AI extraction phase.
